Fixed critical issue where editing or deleting sales (Bon de Commande / BL) 
did not properly update product stock (caisse) in the magasin inventory.

The system now correctly:
1. DECREASES stock when caisse is increased during edit
2. INCREASES stock when caisse is decreased during edit

5. Works with decimal caisse values (e.g., 1.5, 45.3)

================================================================================
INITIAL PROBLEM REPORTED
================================================================================

User Issue:
"When editing a sale and changing the caisse amount, the stock in the 
product page for that magasin does not update. Also, when deleting a sale, 
the stock should go back to the original stock."

Specific Problems:
1. Editing sale: No stock change in Products page
2. Caisse decrease (18→17): Stock not increasing by 1
3. Caisse increase (10→14): Stock decreasing when should (but incorrectly calculated)
4. Admin impersonation: Stock updated for wrong magasin
5. Deleting sale: Stock not returned

================================================================================
ROOT CAUSES IDENTIFIED
================================================================================

Problem #1: Empty sale_items Table
-------------------------------------------
ISSUE:
- Items were saved to sales.items JSONB column ✓
- Items were NOT being saved to sale_items table ✗
- Stock reconciliation only read from sale_items table
- Result: oldItemsRows = EMPTY, newItemsRows = EMPTY
- Calculation: delta = newQty - oldQty = 0 - 0 = 0 (NO CHANGE)

ROOT CAUSE:
Backend received items but failed to insert into sale_items table silently.
No retry logic or fallback existed.

Problem #2: Using Updated sale.store_id Instead of Original
-------------------------------------------
ISSUE:
- Stock reconciliation read sale.store_id AFTER updating it
- If admin changed store_id in request, reconciliation used WRONG store
- Example: Edit sale assigned to Mag1, admin changed to Mag2
  → Stock updated in Mag2 (wrong!) instead of Mag1 (correct!)

ROOT CAUSE:
Code did: 
  1. Update sale in DB (including store_id if provided)
  2. Re-query sale to get store_id
  3. Reconcile stock using updated store_id ✗

Should have been:
  1. Query sale BEFORE update to get original store_id
  2. Update sale in DB
  3. Reconcile stock using ORIGINAL store_id ✓

Problem #3: Old Items Not Available for Comparison
-------------------------------------------
ISSUE:
- When editing a sale, we need OLD caisse values to calculate delta
- Example: Old caisse=18, New caisse=17, Delta should be -1
- But old values weren't being loaded correctly

ROOT CAUSE:
- sale_items table was empty (Problem #1)
- No fallback to extract old values from sale.items JSONB
- Result: oldQty always = 0, calculation was wrong

Problem #4: No Delete/Removal Logic
-------------------------------------------
ISSUE:
- Deleting a sale completely removed it from DB
- Stock was never reversed (given back to inventory)

ROOT CAUSE:
DELETE endpoint only had:
  await supabase.from("sales").delete().eq("id", saleId);
No stock reversal logic existed at all.

Fix #1: Added Fallback Logic for sale_items
-------------------------------------------
Location: PUT /sales/:id handler (line ~8650)

BEFORE:
  const { data: newItemsRows } = await supabase
    .from('sale_items')
    .select('product_id, caisse, quantity')
    .eq('sale_id', saleId);
  // If empty, we fail silently

AFTER (Added 4 fallback strategies):

  1. Try to load from sale_items table (primary)
  2. Retry query after brief delay (handles DB replication lag)
  3. Extract from request body items (if just saved)
  4. Extract from sale.items JSONB (last resort)

Code Pattern:
  let finalNewItems = newItemsRows;
  if (!finalNewItems || finalNewItems.length === 0) {
    // Try retry from DB
    if (oldItemsRows.length > 0) {
      // Got old items, retry new items query
      const retryRows = await supabase.from('sale_items')...;
      finalNewItems = retryRows;
    }
    // Extract from request body
    if (!finalNewItems) {
      finalNewItems = body.items.map(it => ({...}));
    }
    // Extract from JSONB
    if (!finalNewItems) {
      finalNewItems = sale.items.map(it => ({...}));
    }
  }

Result: ✓ Always have items data for reconciliation

Fix #2: Use Original store_id, Not Updated One
-------------------------------------------
Location: PUT /sales/:id handler (line ~8502)

BEFORE:
  const { data: beforeSale } = await supabase
    .from("sales")
    .select("id, store_id, client_id...")  // No items field!
    .eq("id", saleId);
  
  // ... later, after update ...
  
  const { data: saleRow } = await supabase
    .from('sales')
    .select('store_id, sale_number...')  // Re-queries AFTER update
    .eq('id', saleId);
  const storeId = saleRow.store_id;  // WRONG - uses updated value!

AFTER:
  const { data: beforeSale } = await supabase
    .from("sales")
    .select("id, store_id, client_id, items")  // Added items!
    .eq("id", saleId);
  
  // Store the ORIGINAL store_id
  const originalStoreId = beforeSale.store_id;
  
  // ... update happens ...
  
  // Use ORIGINAL store_id for reconciliation
  const storeId = originalStoreId;  // CORRECT!

Result: ✓ Stock always updated for correct magasin

Fix #3: Added Old Items Extraction from JSONB
-------------------------------------------
Location: PUT /sales/:id handler (line ~8750)

NEW CODE:
  let finalOldItems = oldItemsRows || [];
  
  // FALLBACK: Extract from beforeSale.items JSONB
  if ((!finalOldItems || finalOldItems.length === 0) && 
      Array.isArray(beforeSale?.items)) {
    console.warn('[sales PUT] oldItemsRows empty, extracting from beforeSale.items');
    finalOldItems = beforeSale.items.map(it => ({
      product_id: it?.id || it?.product_id,
      caisse: it?.caisse,
      quantity: it?.quantity,
    })).filter(it => it.product_id);
  }
  
  // Use finalOldItems for calculation
  finalOldItems.forEach(it => {
    oldByProduct.set(productId, oldByProduct.get(productId) + caisse);
  });

Why This Works:
- sale.items JSONB is the SNAPSHOT of items at creation time
- These are the OLD caisse values
- We compare: old (from JSONB) vs new (from request)
- Delta = new - old is accurate

Example in Action:
  Original sale items (JSONB): caisse = 18
  User edits to: caisse = 17
  Calculation:
    oldQty = 18 (from JSONB)
    newQty = 17 (from request)
    delta = 17 - 18 = -1
    stock reversal: prev + 1 = prev + 1 ✓ CORRECT!

Fix #4: Added Complete DELETE Handler with Stock Reversal
-------------------------------------------
Location: DELETE /sales/:id handler (line ~9283)

NEW LOGIC (7 steps):

Step 1: Fetch sale before deletion
  - Get store_id (needed to find which magasin's stock to update)
  - Get items (needed to know which products and quantities)
  - Get sale_number (to check if internal doc)

Step 2: Check if normal sale (exclude TRANSFER-, PURCHASE-)
  - Internal docs handled differently
  - Only normal BL sales reverse stock

Step 3: Load sale_items from DB (with fallback to JSONB)
  - Try sale_items table first
  - Fall back to sale.items JSONB if empty

Step 4: Calculate total caisse per product
  - Sum all caisse values for each product
  - Example: ["product_A" caisse=10, "product_A" caisse=4] → 14 total

Step 5: UPDATE store_stocks (ADD caisse back)
  - For each product: new_quantity = old_quantity + reversed_caisse
  - Create entry if doesn't exist

Step 6: Delete the sale
  - Safe to delete now that stock is reversed

Step 7: Log everything
  - Helps debug if something goes wrong

Code Pattern:
  for (const [productId, reverseQty] of reverseByProductId) {
    const current = store_stocks[productId] || 0;
    const newQty = current + reverseQty;  // ADD BACK (opposite of edit)
    await update(store_stocks, newQty);
  }

Result: ✓ Stock fully reversed when sale deleted

TECHNICAL DETAILS
================================================================================

Decimal Handling:
-------------------------------------------
- Caisse supports decimals: 1.5, 45.3, 0.25, etc.
- stored in DB as NUMERIC(12,3) in both sale_items and store_stocks
- toNum() function handles French comma format: "14,5" → 14.5
- round2() function ensures precision: round((n + EPSILON) * 100) / 100

Helper Functions Used:
  toNum(v): Converts string/number to number, handles comma format
  round2(n): Rounds to 2 decimal places with EPSILON for precision
  
Delta Calculation:
  delta = newQty - oldQty
  If delta > 0: Stock DECREASES (sale took more)
  If delta < 0: Stock INCREASES (sale took less)
  
  Next stock = MAX(0, current_stock - delta)
  
  Examples:
    delta = 5: next = current - 5 (decrease by 5)
    delta = -1: next = current - (-1) = current + 1 (increase by 1)

Store Resolution Priority (Edit):
  1. beforeSale.store_id (ORIGINAL - what we use for reconciliation)
  2. Or empty string if not set
  
Store Resolution Priority (Delete):
  1. sale.store_id (primary)
  2. Reverse only for normal sales (not TRANSFER-, PURCHASE-)

Fallback Chain (Items Loading):
  NEW items:
    1. sale_items table
    2. Retry query (DB replication delay)
    3. Request body items
    4. sale.items JSONB
  
  OLD items:
    1. sale_items table (historical)
    2. sale.items JSONB (creation snapshot)

================================================================================
TROUBLESHOOTING GUIDE
================================================================================

If Stock Still Not Updating After Edit:

1. Check browser console for [sales PUT] logs
2. Verify oldItemsRows is populated
3. Check if delta = newQty - oldQty is correct
4. Confirm store_id matches the correct magasin
5. Look for store_stocks table updates in query logs

If Stock Reversal Not Working on Delete:

1. Check browser console for [sales DELETE] logs
2. Verify sale exists and has items
3. Confirm store_id is correct
4. Check if reverseByProductId has products
5. Verify store_stocks update actually happened

If Admin Impersonation Causes Wrong Store:

1. Check that beforeSale.store_id is used (not updated store_id)
2. Verify the ORIGINAL store_id was captured at start of PUT
3. Log should show: storeId: [original_store_uuid] in reconciliation

If Decimal Caisse Values Don't Work:

1. Verify sale_items.caisse is type NUMERIC(12,3)
2. Check store_stocks.quantity is type NUMERIC
3. Ensure toNum() is handling French comma format
